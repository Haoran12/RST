# RST App Design System

## 1. 目标

本文件定义 RST MVP 的应用级界面规范。

Android 端 UI 以 `Tavo` 为主要交互与页面组织参考。RST 需要参照它在聊天体验、浮层关系、消息操作和编辑跳转上的成熟做法，同时保留自己的配色与产品气质。

设计目标：

- Android 竖屏优先，界面交互体验参考 Tavo，但配色保持 RST 自主风格
- 聊天、Lore、角色编辑、日志查看共用同一套视觉语言
- 保留信息密度与可维护性，不因风格化而牺牲可读性
- 让 RST 看起来像独立产品，而不是后台管理页或通用表单 App

---

## 2. 视觉方向

### 2.1 总体气质

视觉基线采用：

- 原生 App 感的界面层级
- 清晰的浮层关系
- 聊天主流程优先的操作组织
- 连续空间感，而不是后台式拼贴白卡
- 配色独立于 Tavo，不复制其紫粉霓虹语言

参考 Tavo 时应优先学习：

- 聊天区、配置区、底部输入区之间的层级关系
- 大圆角与浮层感
- 按钮、标签、切换器的胶囊化表达
- 页面带有情绪氛围，而不是纯工具面板感
- 消息操作与编辑跳转路径足够短

不直接照搬：

- 紫粉霓虹式品牌配色
- 过重的角色立绘依赖
- 装饰性强于信息表达的视觉元素
- 影响表单输入效率的夸张留白

### 2.2 色彩

RST 配色独立设计，不参考 Tavo 的紫粉系。MVP 推荐使用更克制的冷色科技系：

- Background Base: `#0F141B`
- Background Elevated: `#151C24`
- Surface Card: `#1B2430`
- Surface Overlay: `#243140`
- Surface Active: `#2D3D52`
- Border Subtle: `#314154`
- Border Strong: `#4C8DFF`
- Accent Primary: `#2F7CFF`
- Accent Secondary: `#32C7C1`
- Accent Tertiary: `#89A8FF`
- Text Strong: `#F4F7FB`
- Text Secondary: `#C3CDD9`
- Text Muted: `#8C99A8`
- Success: `#43C488`
- Warning: `#F3B24F`
- Error: `#EC6A5E`

使用原则：

- 页面大背景不使用纯黑，采用低饱和蓝灰深底
- 主强调色默认使用 `Accent Primary`
- 危险、private、debug 不共享同一强调色
- 大面积背景仅允许低对比冷色渐变，不使用高噪点纹理影响阅读

### 2.3 光感与背景

为了保留沉浸感但不复制 Tavo 配色，Android 端应默认支持以下背景语言：

- 顶部或右上存在柔和氛围光晕
- 卡片与底部输入条有轻微发光边缘
- 页面背景由 2 至 3 层渐变叠加，而不是单色填充

建议背景结构：

1. `Base gradient`：`#0F141B -> #121A24 -> #101823`
2. `Ambient glow`：在页头附近加入低透明度蓝青径向光
3. `Noise/texture`：仅可选，透明度极低，不应影响文本锐度

### 2.4 圆角、描边与阴影

- 页面主卡片圆角：20dp
- 二级卡片圆角：18dp
- 输入框圆角：22dp
- Tag / Chip 圆角：999dp
- Bottom sheet 顶角：28dp

规则：

- 优先用描边和明暗对比建立层级，其次才是阴影
- 阴影以近距离柔和扩散为主，不做明显 Material 默认投影
- 卡片边框允许使用半透明亮边，营造“浮起来”的感觉

### 2.5 间距

- 页面水平边距：16dp
- 页面顶部安全区后首屏边距：12dp
- 主分组垂直间距：14dp
- 卡片内部间距：14dp
- 紧凑控件间距：8dp
- 聊天消息间距：10dp

## 3. 字体与排版

优先使用 Android 可稳定落地且与整体气质匹配的字体组合：

- 中文正文：`Noto Sans SC`
- 英文与数字：`IBM Plex Sans`
- 等宽文本：`JetBrains Mono`

层级：

| 用途 | 字号 | 字重 | 行高 |
|---|---|---|---|
| 页面标题 | 24 | 700 | 1.15 |
| 重点卡片标题 | 18 | 600 | 1.25 |
| 分区标题 | 16 | 600 | 1.3 |
| 正文 | 15 | 400 | 1.5 |
| 次要信息 | 13 | 500 | 1.4 |
| 标签/胶囊 | 12 | 600 | 1.2 |
| 辅助计数/状态 | 11 | 600 | 1.2 |

规则：

- 标题允许更靠近 Tavo 的“紧凑、偏展示型”风格，但不依赖高饱和霓虹色
- 正文与表单说明必须回到稳定、清晰、对比充足的排版
- 聊天正文避免过细字重
- 长文本输入区与日志区禁止使用装饰性字体

## 4. 导航结构

### 4.1 移动端

采用四个主 Tab：

1. 聊天
2. Lore
3. 设置
4. 日志

导航表现参考 Tavo 的“原生沉浸式入口感”，但配色使用 RST 自己的体系：

- 底部导航使用高圆角容器，可做半透明浮层
- 当前 Tab 使用发光胶囊或高亮描边，不只靠图标变色
- 一级导航固定可见
- 二级编辑页统一全屏 push

不采用：

- Android 端复杂桌面式多栏
- 作为主流程的悬浮小窗编辑

### 4.2 Windows

MVP 仅做基础响应：

- 宽度充足时可切为 `NavigationRail`
- 保留同一配色与层级，不回退成传统亮色桌面后台风
- 页面结构不重排为复杂 IDE 式编辑器

---

## 5. 组件视觉规范

### 5.1 卡片

所有卡片默认采用：

- 深色底
- 细描边
- 轻微背景模糊或半透明感
- 少量内边距而非巨大留白

场景区分：

- 主要内容卡：用于聊天、表单分组、日志摘要
- 信息状态卡：用于 SceneState、Provider 状态、导入提示
- 危险卡：用于删除、private、敏感配置

### 5.2 按钮

按钮必须摆脱传统 Material 扁平蓝按钮观感。

主按钮：

- 胶囊形
- 冷蓝高亮纯色或低对比渐变
- 文本强对比
- 可带轻微外发光

次按钮：

- 深色底 + 描边
- hover / pressed 时抬升亮度

危险按钮：

- 不使用大面积纯红
- 以深底 + 红色描边/文字提示危险性

### 5.3 输入框

输入框参考 Tavo 的底部输入栏与设置面板：

- 深色底
- 大圆角
- 内阴影或轻描边
- 聚焦时边缘高亮，不使用亮蓝默认焦点
- 快捷的引号,括号,MarkDown排版符号输入支持

规则：

- 多行输入框高度充足，支持长文本
- label 优先置于输入框外上方
- 帮助说明使用次级文本，不塞进 placeholder

### 5.4 Tag / Chip / Badge

用于模式、状态、分类、constant、private 提示。

视觉要求：

- 统一胶囊化
- 文字较小但字重更高
- 用色清晰区分，不堆太多图标

建议语义：

- `ST` / `RST`：蓝色高亮
- `constant`：青绿
- `disabled`：低饱和灰蓝
- `private`：琥珀或暖橙
- `error`：暗红

### 5.5 列表项

列表项不做传统“白底一行 + 分割线”。

建议结构：

- 整项作为可点击深色卡片
- 左侧信息块
- 右侧状态或操作入口
- 激活态使用高亮描边或局部光效

## 6. 页面规范

### 6.1 聊天页

聊天页是最需要贴近 Tavo 交互气质的页面，但配色保持 RST 自主方案。

组成：

- 顶部 App Bar：会话名、切换入口、当前模式
- 顶部氛围区：允许极轻量背景光效或封面式区域
- 可选 SceneState 状态卡
- 消息列表
- 底部浮动输入区

聊天页规则：

- 输入区与消息区视觉分层要明确，类似 Tavo 的“悬浮 composer”
- user 与 assistant 气泡明显区分，但避免高对比刺眼纯色
- system/debug 使用独立样式，不与普通对话同级
- streaming 有适度的“生成中”反馈
- 允许消息卡带轻微半透明背景，不使用纯实体白卡

建议：

- assistant 气泡偏 `Surface Overlay`
- user 气泡偏 `Surface Active`
- 输入区使用单独浮层，略高于消息区

#### 6.1.1 顶部栏交互

顶部栏至少包含：

- 返回或会话入口
- 当前 Session 名称
- 模式标记
- 右侧快捷操作入口

交互规则：

- 点击 Session 名称或左侧入口，打开 Session 切换 sheet，而不是跳转到单独列表页
- 右上角提供 `会话设置` 入口，进入当前 Session 的配置页
- 若当前为 `RST` 模式，SceneState 卡片点击后直接进入 Scene 编辑页
- 聊天页到配置页、编辑页统一采用全屏 push，返回时恢复原聊天滚动位置与输入草稿

#### 6.1.2 输入区与发送状态

输入区是聊天页的一级操作中心。

空闲态：

- 多行输入框可扩展高度
- 输入为空或仅空白时，发送按钮禁用
- 发送按钮默认是高亮主按钮

发送态：

- 点击发送后，先本地创建 user 消息，再创建 assistant 占位消息
- 输入框在本地写入成功后清空
- 列表自动滚动到底部
- 发送按钮切换为 `停止`

流式中：

- `停止` 按钮必须始终可见且单手易点
- 若用户当前停留在列表底部附近，流式更新时自动跟随到底部
- 若用户已上滑查看历史，停止自动跟随，并显示“回到底部”悬浮入口

停止后：

- 立即中止 Provider 请求
- 已生成内容保留在当前 assistant 气泡中
- 当前气泡退出 streaming 态，不出现第二条 assistant 最终气泡
- 该条回复应提供 `重新生成` 或 `重试` 入口

#### 6.1.3 消息级操作

消息气泡分为两层操作：

- 直显快捷条：常用高频操作，直接显示在气泡底部
- 展开菜单：低频或危险操作，通过 `...` 或长按打开 bottom sheet

assistant 消息直显快捷条固定为：

- 回溯
- 重新生成
- 改写
- 复制

user 消息直显快捷条默认显示：

- 回溯
- 改写
- 复制

assistant / user 消息展开菜单统一包含：

- 设置可见性
- 删除
- AI代答

error 消息额外直显：

- 重试

规则：

- 直显快捷条默认最多显示 4 个动作，采用轻量文字按钮或小胶囊按钮，不使用大图标矩阵
- 同一轮消息优先以“轮次”为单位理解操作，而不是孤立单条消息
- `删除` 默认执行“删除本轮”，优先删除当前 user 消息及其对应 assistant 消息，避免留下孤立半轮对话
- 删除历史消息前必须二次确认，并提示“后续上下文可能变化”
- `重新生成` 与 `重试` 默认复用同一轮 assistant 占位消息，不额外追加一条最终回复
- 消息删除成功后，列表应立即移除对应气泡，并保持当前位置尽量稳定
- 被隐藏的消息在列表中仍可见，但视觉上应明确标记为“已隐藏”

动作语义：

- `回溯`：回到当前消息所在轮次，删除该轮之后的所有消息；若当前定位在 assistant 气泡，则按其对应 user 轮次作为回溯锚点
- `重新生成`：复用当前轮 user 输入，覆盖式生成 assistant 回复
- `改写`：把当前轮 user 文本加载到编辑态 composer；若从 assistant 气泡触发，则先定位到该轮对应 user 文本再进入编辑态
- `复制`：复制当前气泡纯文本内容
- `设置可见性`：在 `显示` 与 `隐藏` 间切换；隐藏后该消息在提示词组装、上下文截断计算与 RST 调度中一律忽略
- `删除`：删除当前轮消息，必要时支持“只删当前条”作为二级选项，但默认不暴露
- `AI代答`：基于当前上下文为“下一条 user 发言”生成草稿，结果先进入 composer 供用户确认，不直接发送

视觉要求：

- 隐藏消息显示低对比样式，并带 `已隐藏` badge
- 当前被回溯锚定的消息在确认前应有短暂高亮反馈
- `AI代答` 属于能力型操作，放在展开菜单，不与高频生成动作抢主视觉层级

#### 6.1.4 失败、重试与重新生成

失败态要可见但不应打断阅读。

规则：

- assistant 消息失败时，在气泡底部显示内联 `重试` 次按钮
- 用户主动停止后，优先显示 `重新生成`，而不是红色错误样式
- `重试` 复用原 user 输入，不要求用户把文本重新填回输入框
- `重新生成` 用于已经有内容但用户希望覆盖本轮 assistant 回复的场景
- 若开始重试或重新生成，应复用该轮 assistant 消息卡片并清空旧的局部错误提示
- 若消息当前为隐藏状态，`重试`、`重新生成`、`AI代答` 在执行前应先提示其不会把隐藏消息带入上下文

#### 6.1.5 草稿、滚动与返回恢复

- 用户切到 `Lore`、`设置`、`日志` 或全屏编辑页后，聊天输入草稿必须保留
- 从配置页、Scene 页、Character 页返回聊天页时，应恢复先前滚动位置
- 若用户离开聊天页时仍在流式接收，返回后应继续展示同一条正在生成的消息
- 若应用重启后上一轮流式已异常中断，应在该消息上展示可重试状态

### 6.2 Session 列表页

列表项至少展示：

- Session 名称
- 模式标记 `ST` / `RST`
- 最近更新时间

交互要求：

- 支持创建、重命名、删除
- 删除前必须二次确认
- 空状态提供“新建 Session”主按钮
- 页面进入时触发一次懒扫描，自动发现离线拷贝导入的 Session
- 若存在重复 ID 或损坏对象，列表页应展示非阻塞告警

视觉建议：

- 列表顶部允许有情绪化标题区
- 每个 Session 卡更接近“会话入口卡”而不是数据库行

### 6.3 Lore 页

Lore 页保持工具效率，但视觉仍需属于同一系统。

包含：

- ST World Book 选择或切换入口
- 搜索框
- 分类筛选
- constant 标记
- 启用/禁用状态
- 新建条目按钮

每条列表项至少展示：

- 名称
- 分类
- tags 摘要
- constant / disabled 状态

规则：

- 搜索与筛选区域可做吸顶胶囊控件
- 列表项支持一眼看出状态，不依赖进入详情页
- 不做复杂树结构

### 6.4 World Book 列表页

列表项至少展示：

- 世界书名称
- 最近更新时间

交互要求：

- 支持创建、重命名、删除
- 删除前提示“可能影响仍绑定该世界书的 ST Session”
- 页面进入时触发一次懒扫描，自动发现离线拷贝导入的世界书

### 6.5 Character 编辑页

固定三个 Tab：

1. 基础
2. 形态
3. 关系

规则：

- Tab 采用胶囊分段控件，而不是传统细下划线 Tab
- 每个 Tab 控制表单长度，避免单页过长滚动
- `public_*` 与 `private_*` 视觉上明确分组
- private 区块需要独立背景或描边，避免误填

### 6.6 Scene 编辑页

字段：

- 时间
- 地点
- 在场人物
- 备注

交互要求：

- 保存时提交完整 SceneState
- 未保存修改离开时提示
- 不做自动 merge 或隐式局部保存

视觉建议：

- 该页可用更强的“状态面板”感
- 适合加入卡片式字段分组，而不是连续长表单

### 6.7 设置页

按信息架构分区：

- Session
- API Config
- Preset
- Appearance

规则：

- 打开 API Config / Preset / Appearance 对应列表时，系统应懒扫描各自目录
- 符合格式的离线拷贝配置文件应自动生效
- 无法容纳的配置文件应显示错误提示，但不阻塞其他配置使用
- 避免所有设置堆成单页长表单

设置页应接近 Tavo 的“配置中心”观感，但不复用其品牌配色：

- 分组明确
- 卡片层次强
- 风格统一但不过分花哨

### 6.8 日志页

日志页要呈现真实的请求与响应信息。

列表项展示：

- 状态
- 时间
- model
- duration
- token 数

详情页展示：

- 请求摘要
- 响应摘要
- 截断标记
- 脱敏标记

视觉规则：

- 日志页整体仍维持主主题深色系统
- 代码块、JSON 预览可进一步降低亮度并使用等宽字体
- 错误日志需更醒目，但不破坏全局配色

### 6.9 页面跳转与编辑流

聊天页与配置页、编辑页之间的跳转必须尽量短链路，不让用户反复回到首页找入口。

推荐入口：

- 聊天页左上角：进入当前 Session 设置
- SceneState 卡片：进入 Scene 编辑页
- 在场人物 Chip：进入对应 Character 编辑页
- 底部 Tab `Lore`：查看当前模式关联的 Lore / World Book
- 底部 Tab `设置`：进入全局配置中心
- 底部 Tab `日志`：进入请求日志页

导航规则：

- 编辑页统一全屏 push
- 底部 Tab 切换属于一级导航，保留各自页面状态
- 从聊天页进入任何编辑页后，返回应回到原 Session 的聊天页，而不是重置到列表首页
- 若当前编辑页存在未保存修改，返回时必须拦截并提示
- 从聊天页跳转到设置页时，当前 Session 上下文应保留，返回后不重新创建页面状态

---

## 7. 表单设计规范

### 7.1 通用规则

- 单列表单优先
- 每组字段前显示分组标题与一句短说明
- 必填与可选明确区分
- 长文本字段提供足够高度
- 输入错误时提供明确的内联提示，而不是仅靠红色边框
### 7.2 危险字段

以下字段需要更强提示：

- API key
- private_profile
- private_state
- destructive actions

呈现方式：

- 使用独立危险卡或醒目 badge
- 不只靠红字提示

### 7.3 保存策略

MVP 推荐：

- 编辑页底部固定保存按钮
- 未保存修改离开时提示
- 不采用过多自动保存，降低误写风险
- 保存按钮应始终具有高识别度，符合全局主按钮样式

---

## 8. 状态与反馈

### 8.1 Loading

- 首屏骨架屏仅用于列表和消息流
- 表单页优先用局部 loading，不阻塞整个页面
- 骨架屏颜色与主题一致，避免默认灰白骨架破坏氛围

### 8.2 Empty State

需要单独设计：

- 无 Session
- 无 Lore
- 无 Character
- 无日志

规则：

- 空状态必须带下一步操作按钮
- 允许少量插画式氛围元素，但不依赖复杂资产

### 8.3 Error State

- 网络错误可重试
- 存储错误明确提示“本地保存失败”
- 流式响应中断时保留已生成文本，并允许重新发送
- 错误提示优先做成内联卡片或顶部提醒，不滥用系统 Toast

---

## 9. Scene 与可见性呈现

### 9.1 SceneState

聊天页中的 SceneState 只展示：

- 时间
- 地点
- 在场人物

详细备注进入单独编辑页。

SceneState 摘要卡应有“当前场景面板”感，与普通消息区分。

### 9.2 Public / Private 分层

人物编辑页必须明确区分：

- Public Profile
- Private Profile
- Public State
- Private State

禁止把 private 内容和 public 内容放在同一无标题文本块中。

建议：

- private 分区使用更偏暖色的高亮描边
- public 分区保持标准卡片样式

## 11. 优先设计

MVP 的设计优先级是：

1. 先建立 Tavo 参考下的整体气质与层级
2. 保证聊天主流程顺手
3. 保证表单编辑与排错效率
4. 所有页面共用统一组件语言
5. 在后续扩展前保持可维护
